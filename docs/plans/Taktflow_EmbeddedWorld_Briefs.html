Why we are talking to TÜV
We built a pipeline that produces ISO 26262 safety evidence automatically — traceability from HARA assumptions to measured hardware results, on every commit. Before we present this to customers as a path toward functional safety certification, we need to understand from TÜV's perspective what that evidence is actually worth — and where the gaps are between what we produce and what a safety case actually requires.
The core safety claim
Every HARA assumption, every TSC assumption, every software safety requirement carries forward an estimate about the layer below it. Those estimates are often wrong.

Our pipeline makes those assumptions continuously testable. A safety goal that says "safe state reachable in 10ms" is tested against real hardware on every commit. The measured result — 9.8ms — is timestamped and linked to the safety goal by requirement ID.

The assumption is no longer an estimate. It is a measurement. Updated on every build.
What we produce for ASIL D evidence
Structural coverage: MC/DC via LLVM per commit. Statement, branch, and condition coverage.

Requirements-based testing: every test linked to a SW safety requirement via REQ-ID. JUnit XML as evidence.

Fault injection: HIL relay boards inject hardware faults. Measured reaction time vs safety goal.

E2E verification: AUTOSAR E2E profile tested in SIL for single-bit and burst error detection.

Traceability chain: HARA → TSC → SW safety req → test case → SIL/PIL/HIL result. All linked.
What we claim to satisfy
ISO 26262-6: software unit design and implementation

ISO 26262-6 Table 10: MC/DC for ASIL D

ISO 26262-6 Table 11: requirements-based test

ISO 26262-4: technical safety concept verification

ISO 26262-9: ASIL decomposition traceability
Known limitations
No tool qualification (ISO 26262-8:11) — open source tools not formally qualified

No independence between developer and tester in single-person setup

HIL not formally calibrated — timing measurements are indicative, not certified

Safety case structure not yet formally documented per ISO 26262-2
What changes this claim
Tool qualification of open-source tools via TÜV assessment

Two-person review — developer + independent safety reviewer

Calibrated HIL timing with traceability to measurement equipment

Formal safety case document linking all evidence artifacts
